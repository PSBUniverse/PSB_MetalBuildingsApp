## **Batch Edit Mode – Row Behavior Fix (I already handled the action button behavior based on edit state and record state, this is expanded behaviour of the batch edit mode)**

### **Current Issue (Fix This)**

* When a user clicks **Deactivate**, the system:

  * Creates a staged batch change ✅
  * **Disables the entire row ❌ (this is wrong)**

## **Key Principle**

```Staging ≠ locking.```

The row should stay alive and reversible at all times until “Save Batch” is triggered.

---

## **Required Behavior**

### **1. Do NOT disable the row**

* Rows must remain **fully interactive** even after:

  * Deactivate is clicked
  * Edit is clicked

---

### **2. Replace actions instead of disabling**

#### **Deactivate → Restore**

* When user clicks **Deactivate**:

  * Stage the change (batch edit record)
  * Keep row active (no disabled state)
  * Replace **Deactivate button → Restore button**

#### **Edit → Cancel**

* When user clicks **Edit**:

  * Enable inline editing
  * Keep row active (no disabled state)
  * Replace **Edit button → Cancel button**

---

### **3. No “disabled row” state in batch mode**

* Completely remove any logic that:

  * Adds `disabled`, `opacity`, or `pointer-events: none`
  * Prevents interaction with the row

If it’s staged, it should still feel editable—not frozen. But keep the highlight trigger/markers/tag triggers to know that "this" row has been edited/updated/changed.
---

## **Correct Interaction Model**

| Action Clicked | What Happens                               |
| -------------- | ------------------------------------------ |
| Deactivate     | Stage change → Show **Restore**            |
| Restore        | Remove staged change → Show **Deactivate** |
| Edit           | Enter edit mode → Show **Cancel**   |
| Cancel         | Exit edit mode → Revert changes            |
